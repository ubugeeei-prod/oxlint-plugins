# @oxlint-plugins/oxlint-plugin-postgresql

Rust-backed Oxlint port of [`eslint-plugin-postgresql`](https://github.com/baseballyama/eslint-plugin-postgresql) — lints SQL.

> **Status: in progress.** The 89 upstream rules of `eslint-plugin-postgresql@0.22.1` are tracked in [`status.json`](../../status.json); rules are ported one per PR. See [`docs/port-targets/eslint-plugin-postgresql.md`](../../docs/port-targets/eslint-plugin-postgresql.md) for the canonical rule inventory.

## Architecture

SQL is parsed by **libpg_query (PostgreSQL 17)** — the actual PostgreSQL parser — through the [`pg_query`](https://crates.io/crates/pg_query) crate, statically linked into the native addon. The crate's C `pg_query_parse` JSON entry point is called directly so the parse tree keeps PostgreSQL's original node field names (`SelectStmt`, `targetList`, `ResTarget`, `A_Star`, …), exactly as upstream's [`postgresql-eslint-parser`](https://github.com/baseballyama/postgresql-eslint-parser) (which wraps the PG17 `@libpg-query/parser` WASM build) emits them.

The parser's location pipeline (tokenizer + node-span resolution) and every rule are ported to Rust in [`crates/postgresql`](../../crates/postgresql) and exposed through NAPI-RS. This JavaScript layer is only the Oxlint/ESLint-compat adapter and native API.

Oxlint does not currently route `.sql` files to JavaScript plugins, so this package's CLI integration is disabled in status metadata (mirroring the `@eslint/json` and `@eslint/markdown` ports); the native API and adapter are exercised by the parity test suite.

## Supported platforms

libpg_query does not build under the Windows MSVC toolchain, so — unlike the other ports — this package targets Linux (gnu, x64/arm64) and macOS (x64/arm64) only.

## Test parity

Upstream authors its rule tests as on-disk fixtures (`tests/fixtures/<rule>/{valid,invalid}/*.sql` plus sibling `.yaml` expectations generated by its own RuleTester replay). `pnpm run port:tests:postgresql` captures those fixtures verbatim into [`test/fixtures/*.json`](./test); the Vitest suite replays them against this plugin and enforces exact parity per rule (see `test/parity.json`).
